1. What the Project Is Supposed to Do

Correct Requirement:

A farmer is already on a specific symptom page (e.g., "Leaf Rust" or "Galls"). They are looking at reference images of that symptom and wondering: "Does my plant have this?"

They capture a photo of their crop. The system compares that photo only against that one symptom's reference gallery and returns a single similarity percentage.

  • Input: One uploaded image + one symptom ID (the page they're on)
  • Output: "Your image is 73% similar to Galls"
  • Decision support: ≥80% = "Take Action", 60-79% = "Monitor", <60% = "Low Concern"
Correct Flow

Farmer is on
"Galls" page → Captures
field photo → Compare ONLY
against Galls gallery → Result: "73% match
to Galls — Monitor"

This is a 1-to-1 confirmation task: "Does my image match this symptom?" — not a search across all symptoms.

2. What the Current Code Actually Does

Current Implementation (Wrong):

The POST /compare endpoint accepts an image with no symptom ID. It compares the uploaded image against every symptom's embeddings in the entire gallery, ranks all of them, and returns the top 5 matches across all symptoms.

Current (Incorrect) Flow

User uploads
image only → Compare against
ALL 10 symptoms → Return top 5 from
any symptom → "87% Leaf Rust,
73% Galls, 61%..."

What the current /compare endpoint does (line by line):

# routes.py — POST /compare
# 1. Accepts file upload with NO symptom_id parameter
# 2. Loops through EVERY symptom in GALLERY (all 10):
for symptom, embeddings in GALLERY.items():
    for ref_emb in embeddings:
        sim = cosine_similarity(query_emb, ref_emb)
        scores.append((sim, symptom))

# 3. Sorts ALL scores, takes top 5 across ALL symptoms:
scores.sort(reverse=True)
top_matches = scores[:5]

# 4. Returns multi-symptom results — NOT what was asked for

This is essentially a "What disease does my plant have?" engine — a fundamentally different product from "Does my plant match this specific symptom?".

3. The Core Requirement Mismatch — Detailed Breakdown

Aspect What's Needed What's Built
API Input Image file + symptom_id Image file only (no symptom_id)
Comparison scope Compare against one symptom's gallery Compares against all 10 symptoms
Output Single similarity score for the target symptom Top 5 ranked matches from any symptom
Use case Confirmation: "Is this Galls?" Discovery: "What disease is this?"
User context Already on a symptom page, needs guidance No context — open-ended search
Evaluation metric Verification accuracy (true match vs false match) Precision@K / Recall@K (retrieval metrics)
Fine-tuning objective Better same-vs-different separation for each symptom Better global ranking across all symptoms
Impact: The entire evaluation pipeline (Precision@5, Recall@5, baseline vs fine-tuned comparisons) measures retrieval quality — "Did the correct symptom appear in top-K?" This is the wrong metric for a verification task. The correct metrics would be verification accuracy, AUC-ROC (can we separate match from non-match?), and threshold calibration (at what similarity % is the system reliable?).

5. Technology Review — Current Stack vs Market Alternatives

5.1 — Embedding Model

Library / ModelWhat It DoesProsConsVerdict for Our Use Case
DINOv2 ViT-S/14
Current		 Self-supervised vision transformer from Meta. 384-dim embeddings. 21M params. Excellent visual feature extraction; strong on texture/shape; no labels needed for pretraining Generic features — doesn't understand plant diseases without fine-tuning; 224px input is low for detail Good choice — keep it. Best general-purpose vision encoder available. Fine-tuning addresses domain gap.
DINOv2 ViT-B/14 Larger variant. 768-dim embeddings. 86M params. More capacity for subtle disease features; better accuracy on fine-grained tasks 4x heavier than ViT-S; slower inference Consider upgrading if ViT-S accuracy is insufficient. Test before committing — diminishing returns on small datasets.
CLIP (OpenAI) Vision-language model. Can match images to text descriptions. Can do zero-shot: "Is this leaf rust?" without any training; understands semantic concepts Weaker on fine-grained visual similarity than DINOv2; 151M+ params; optimized for text-image not image-image Not recommended as primary. DINOv2 outperforms CLIP on visual similarity tasks. CLIP is better for captioning.
PlantCLIP / PlantNet models Domain-specific models pre-trained on plant/disease datasets. Already understands plant features; less fine-tuning needed Fewer available; less community support; may not cover crop diseases specifically Worth investigating as a supplementary comparison, but DINOv2 + fine-tuning is the stronger approach with our data.
EfficientNet-B0/B3 CNN-based classifier. Lightweight. Very fast; small model size; good for mobile deployment CNNs are weaker than ViTs on texture/shape similarity; needs classification head, less flexible Not recommended. ViTs are superior for similarity-based tasks. Would be a downgrade.

5.2 — Similarity Search

LibraryWhat It DoesCurrent UsageVerdict
NumPy cosine similarity
Current		 Manual dot product / norm calculation in a Python loop Loops through every embedding one by one in routes.py and similarity.py Inefficient. O(N) Python loop per query. Acceptable for 1,639 images, will break at scale.
FAISS (Facebook) Optimized vector similarity search. GPU support. Billion-scale. Listed in Requirements.txt but NEVER used in any code file. Should be used. Already a dependency. Replace NumPy loops with FAISS IndexFlatIP for exact cosine search — 10-100x faster.
ChromaDB / Qdrant Vector databases with persistence, filtering, metadata. Not used Overkill for current scale (1,639 images). FAISS is sufficient. Consider if gallery grows to 50K+.
scikit-learn cosine_similarity Vectorized cosine similarity on matrices. Listed in Requirements.txt but not used Quick improvement over manual loops. One-liner: cosine_similarity(query.reshape(1,-1), gallery). But FAISS is better.

5.3 — Fine-Tuning / Training

ApproachWhat It DoesCurrent UsageVerdict
Triplet Loss
Current		 Pushes anchor closer to positive, farther from negative. Margin = 0.3. Used with random triplet sampling in fine_tune.py Works, but random sampling is inefficient. Many "easy" triplets contribute zero gradient. Needs hard-negative mining.
ArcFace / CosFace Loss Angular margin loss. Specifically designed for verification tasks ("is this the same class?"). Not used Best fit for our corrected requirement (1-to-1 verification). Used in face verification — directly analogous to "does this image match this symptom?"
Supervised Contrastive Loss (SupCon) Pulls all same-class samples together, pushes different classes apart. More stable than triplet. Not used Strong alternative to triplet loss. Better gradient signal from all pairs in a batch, not just triplets.
pytorch-metric-learning Library with 20+ loss functions, miners (hard negative), samplers. Not used Strongly recommended. Provides ArcFace, hard-negative mining, and proper evaluation — replaces most custom training code.

6. Architecture Issues & Technical Debt

Issue 1: API Uses Baseline Model, Not Fine-Tuned Critical

After spending all of Week 2 on fine-tuning (achieving +27% precision improvement), the API endpoint routes.py:17 still loads the base DINOv2 model:

MODEL = load_dinov2()   # ← loads raw DINOv2, NOT the fine-tuned CropSimilarityModel

The fine-tuned model (data/embeddings/fine_tuned_model.pt) is only used in scripts/evaluate.py. All API responses use the inferior baseline model.

Issue 2: Embeddings Were Built with Baseline, But Fine-Tuned Model Projects to Different Space Critical

The gallery embeddings in data/embeddings/ are 384-dim vectors from baseline DINOv2. The fine-tuned model outputs 128-dim vectors (after the projection head). These are incompatible. You cannot compare a 128-dim fine-tuned query against 384-dim baseline gallery embeddings. If the API is switched to use the fine-tuned model, the entire gallery must be re-embedded.

Issue 3: Image Transform Defined in 4 Separate Places Debt

The identical Resize(256) → CenterCrop(224) → Normalize transform is copy-pasted in:

  • src/model/encoder.py:11
  • src/model/fine_tune.py:74
  • src/api/routes.py:33
  • scripts/evaluate.py:16

Any change to preprocessing must be made in all 4 places or results become silently inconsistent.

Issue 4: Symptom Naming Inconsistency Debt

Folder names in data/raw/ use spaces ("Burnt appearance"), while data/embeddings/ uses underscores ("Burnt_appearance"). Code works around this with .replace(" ", "_") in multiple places, but the API's /gallery/{symptom_id} endpoint requires the caller to guess the naming convention. This will cause bugs when integrating with the mobile app.

Issue 5: No Configuration File Debt

config/config.yaml is empty. All values are hardcoded: thresholds (80/60), model name ("dinov2_vits14"), image size (224), embedding dims (384/128), batch size (16), learning rates, etc. Changing any parameter requires editing source code.

Issue 6: No Authentication on API Production

All endpoints are publicly accessible. In production, the /compare endpoint (which runs GPU inference) is vulnerable to abuse. Needs integration with the Strapi backend's auth or at minimum an API key.

7. What's Working Well

The intern's work has several genuinely solid foundations worth preserving:

  • DINOv2 choice is correct — it's the best available vision encoder for this type of similarity task. Good research decision.
  • Fine-tuning pipeline works — two-phase training with checkpoint resume is well-engineered. The +27% precision gain validates the approach.
  • Image validation is thorough — blur detection (Laplacian variance), brightness checks, format validation, corruption checks. This is production-quality.
  • Project structure is clean — clear separation of model, data, API, scripts, and tests. Good directory organization.
  • FastAPI with Pydantic schemas — typed responses, auto-generated Swagger docs. Correct framework choice.
  • TripletDataset — correct implementation of triplet sampling with transforms. Handles class imbalance implicitly.

8.2 — Corrected Comparison Logic

# Instead of comparing against ALL symptoms:
def compare_against_symptom(query_embedding, symptom_id, gallery):
    """Compare query image against ONE symptom's reference gallery."""

    symptom_embeddings = gallery[symptom_id]  # Only this symptom

    # Compute similarity against each reference image in this symptom
    similarities = faiss_index.search(query_embedding, k=len(symptom_embeddings))

    # Aggregate: use MEAN of top-N similarities (not just max)
    # This is more robust than a single best-match score
    top_n = min(5, len(symptom_embeddings))
    avg_similarity = np.mean(sorted(similarities, reverse=True)[:top_n])

    return {
        "similarity_pct": round(avg_similarity, 1),
        "recommendation": get_recommendation(avg_similarity)
    }

8.3 — Corrected Evaluation Approach

Current (Retrieval Metrics)Needed (Verification Metrics)
Precision@5: "Of top 5 results, how many are correct symptom?" Verification Accuracy: "When I compare image X against symptom Y, does the score correctly indicate match/no-match?"
Recall@5: "Of all correct images, how many appeared in top 5?" AUC-ROC: "How well does the similarity score separate true matches from false matches across all thresholds?"
— Threshold Calibration: "At 80% similarity, what % of the time is it actually the correct symptom?"
— False Positive Rate: "When comparing a Galls image against the Leaf Rust gallery, how often does it incorrectly score above 80%?"

Bottom Line

The foundation is sound — DINOv2, FastAPI, fine-tuning pipeline, image validation are all good choices. But the core product logic is solving the wrong problem. The system searches across all symptoms when it should verify against one. Items #1-4 (P0) fix this fundamental mismatch. Everything else is optimization on a corrected foundation.
